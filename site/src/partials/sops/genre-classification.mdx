## Constraints

- **Taxonomy is compiled.** The alias map in `genre.rs` cannot be changed at runtime. If the user disagrees with a mapping, the agent works around it by using the user's preferred genre directly.
- **No auto-tagging.** Every genre change requires explicit human approval.
- **Cache-first.** Classification runs from cached data. If cache is empty for a track, flag as insufficient evidence тАФ do not trigger enrichment mid-classification.
- **XML export only.** No direct DB writes. All changes flow through `update_tracks` тЖТ `preview_changes` тЖТ `write_xml`.

## Prerequisites

```
cache_coverage(has_genre=false)
```

All providers must be at 100% before proceeding. If not, hydrate first:

```
analyze_audio_batch(has_genre=false, skip_cached=true, max_tracks=200)
enrich_tracks(has_genre=false, skip_cached=true, providers=["discogs"], max_tracks=50)
enrich_tracks(has_genre=false, skip_cached=true, providers=["beatport"], max_tracks=50)
```

Repeat until all providers reach 100%. Report progress between batches.

## Steps

### 0. Metadata quality pre-check

```
search_tracks(has_genre=false)
```

Scan results for tracks with empty `artist` or `title` тАФ these will fail enrichment lookups. If found, present to user with options: fix metadata, skip these, or proceed anyway.

If no tracks have missing fields, skip silently.

### 1. Review taxonomy

```
get_genre_taxonomy()
suggest_normalizations()
```

Present alias mappings to user. For each debatable alias (marked in result):
- **Accept** тАФ use canonical form
- **Override** тАФ user specifies preferred genre
- **Skip** тАФ exclude from batch classification

For unknown genres (not in taxonomy and not an alias): collect their track IDs and include them in Step 2 classification batches. Do NOT ask the user to manually map without evidence.

Store overrides in working memory for the session.

### 1b. Stage alias normalizations

Non-debatable alias mappings (e.g. `Hip-Hop` тЖТ `Hip Hop`, `DnB` тЖТ `Drum & Bass`) can be staged immediately after user confirms:

```
update_tracks(changes=[{"track_id": "...", "genre": "..."}])
```

### 2. Classify tracks

Classify both ungenred tracks and unknown-genre tracks (from Step 1) through the same evidence-based pipeline.

```
resolve_tracks_data(has_genre=false, max_tracks=50, format="classification")
resolve_tracks_data(track_ids=[...unknown_ids...], format="classification")
```

For each track, apply the decision tree (below).

Present grouped by confidence: `# ARTIST тАФ TITLE тЖТ SUGGESTED (evidence summary)`

Ask user: "approve all / approve high only / reject # / change # Genre / details # / skip"

### 3. Stage approved changes

```
update_tracks(changes=[{"track_id": "...", "genre": "..."}])
```

Repeat Step 2 for remaining tracks (track processed IDs to avoid re-processing).

**Batch tracking:** Initialize `classified_ids` list. After each batch, add ALL track IDs regardless of outcome. For subsequent batches, pass remaining unprocessed IDs. Stop when none remain.

To audit tracks that already have genres, see the [Genre Audit SOP](/workflows/genre-audit/).

### 4. Export

```
preview_changes()
```

Ask user: "Export these changes to XML?"

```
write_xml()
```

Report output path. Remind user to import into Rekordbox.
