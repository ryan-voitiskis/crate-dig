## Constraints

- **Taxonomy is compiled.** The alias map in `genre.rs` cannot be changed at runtime. If the user disagrees with a mapping, the agent works around it by using the user's preferred genre directly.
- **No auto-tagging.** Every genre change requires explicit human approval.
- **Cache-first.** Classification runs from cached data. If cache is empty for a track, flag as insufficient evidence — do not trigger enrichment mid-classification.
- **XML export only.** No direct DB writes. All changes flow through `update_tracks` → `preview_changes` → `write_xml`.

## Prerequisites

```
cache_coverage(has_genre=false)
```

All providers must be at 100% before proceeding. If not, hydrate first:

```
analyze_audio_batch(has_genre=false, skip_cached=true, max_tracks=200)
enrich_tracks(has_genre=false, skip_cached=true, providers=["discogs"], max_tracks=50)
enrich_tracks(has_genre=false, skip_cached=true, providers=["beatport"], max_tracks=50)
```

Repeat until all providers reach 100%. Report progress between batches.

## Steps

### 0. Metadata quality pre-check

```
search_tracks(has_genre=false)
```

Scan results for tracks with empty `artist` or `title` — these will fail enrichment lookups. If found, present to user with options: fix metadata, skip these, or proceed anyway.

If no tracks have missing fields, skip silently.

### 1. Review taxonomy

```
get_genre_taxonomy()
suggest_normalizations()
```

Present alias mappings to user. For each debatable alias (marked in result):
- **Accept** — use canonical form
- **Override** — user specifies preferred genre
- **Skip** — exclude from batch classification

For unknown genres (not in taxonomy and not an alias): collect their track IDs and include them in Step 2 classification batches. Do NOT ask the user to manually map without evidence.

Store overrides in working memory for the session.

### 1b. Stage alias normalizations

Non-debatable alias mappings (e.g. `Hip-Hop` → `Hip Hop`, `DnB` → `Drum & Bass`) can be staged immediately after user confirms:

```
update_tracks(changes=[{"track_id": "...", "genre": "..."}])
```

### 2. Classify tracks

Classify both ungenred tracks and unknown-genre tracks (from Step 1) through the same evidence-based pipeline.

```
resolve_tracks_data(has_genre=false, max_tracks=50, format="classification")
resolve_tracks_data(track_ids=[...unknown_ids...], format="classification")
```

For each track, apply the decision tree (below).

Present grouped by confidence: `# ARTIST — TITLE → SUGGESTED (evidence summary)`

Ask user: "approve all / approve high only / reject # / change # Genre / details # / skip"

### 3. Stage approved changes

```
update_tracks(changes=[{"track_id": "...", "genre": "..."}])
```

Repeat Step 2 for remaining tracks (track processed IDs to avoid re-processing).

**Batch tracking:** Initialize `classified_ids` list. After each batch, add ALL track IDs regardless of outcome. For subsequent batches, pass remaining unprocessed IDs. Stop when none remain.

### 4. Review conflicts (optional)

Only if user requests. Query `resolve_tracks_data(has_genre=true, format="classification")` and filter for tracks where evidence disagrees with current genre.

Present conflicts and key mismatches in separate sections.

### 5. Export

```
preview_changes()
```

Ask user: "Export these changes to XML?"

```
write_xml()
```

Report output path. Remind user to import into Rekordbox.

## Decision Tree

Apply for every track during classification.

### Inputs (from `resolve_tracks_data`, format `classification`)

- `current_genre` (may be empty)
- `current_genre_canonical` (may be null)
- `discogs_mapped_genres[]` — each has `genre` and `style_count` (may be null)
- `beatport_mapped_genre` (single canonical value or null)
- `beatport_genre_raw` (original Beatport subgenre string, e.g. "Melodic House & Techno")
- `audio.danceability`, `audio.dynamic_complexity`, `audio.rhythm_regularity`, `audio.spectral_centroid_mean`, `audio.stratum_bpm`, `audio.rekordbox_bpm`, `audio.bpm_agreement`
- Session overrides from Step 1

### Step A: Gather mapped genres

1. From `discogs_mapped_genres`, collect `genre` values, deduplicate → `discogs_genres` (set).
2. `beatport_mapped_genre` → `beatport_genre` (single value or null).
3. If `current_genre` is non-empty, use `current_genre_canonical` if available, else `current_genre` as-is → `current`.

### Step B: Audio screening

Compute an audio profile **before** evaluating enrichment. Audio features establish constraints that enrichment cannot override.

#### B.1 Energy bucket

| danceability | Bucket |
|-------------|--------|
| < 1.0 | `non_dancefloor` |
| 1.0–1.5 | `low_energy` |
| 1.5–2.5 | `dancefloor` |
| > 2.5 | `high_energy` |

#### B.2 Character flags

Compute all that apply:

| Condition | Flag |
|-----------|------|
| dynamic_complexity > 10 | `ambient` |
| dynamic_complexity > 5 | `atmospheric` |
| rhythm_regularity < 0.5 | `broken` |
| rhythm_regularity 0.5–0.8 | `irregular` |
| BPM > 155 | `fast` |
| BPM < 115 | `slow` |

> **Caveat — rhythm_regularity and production style:** Lo-fi, dub, and heavily reverbed productions can register misleadingly low rhythm_regularity because reverb wash smears the rhythmic signal. When `atmospheric` is also set, treat `broken`/`irregular` with lower confidence — the track may actually be straight 4/4.

#### B.3 Audio vetoes

Hard constraints that override enrichment:

- `non_dancefloor` + `ambient` → **Ambient**. Reject any dance-floor genre. Confidence = **medium**, note "audio-vetoed".
- `non_dancefloor` + `slow` → genre must come from the **Downtempo family**. If enrichment suggests a dance genre, ignore it and infer from Downtempo family using enrichment hints. Confidence = **low**.
- `non_dancefloor` + neither `ambient` nor `slow` → genre must come from **Downtempo or Other family**. Flag dance genres as conflicts.
- `fast` + danceability > 1.0 → genre must come from the **Bass family**. Flag non-Bass enrichment as conflict.
- `low_energy` + `atmospheric` + all enrichment genres are dance-floor (House/Techno/Bass) → **likely misclassified**. Prefer Downtempo family. Confidence = **low**, note "audio suggests non-dancefloor".

If an audio veto fires, skip to Step E with the vetoed/constrained genre.

### Step C: Find consensus

Only reached if no audio veto fired.

4. **Full consensus:** `beatport_genre` exactly equals an entry in `discogs_genres` → confidence = **high**.

5. **Partial consensus (Discogs only):** `beatport_genre` is null, `discogs_genres` has exactly one entry → confidence = **medium**.

6. **Partial consensus (Beatport only):** `discogs_genres` is empty, `beatport_genre` is non-null → confidence = **medium**.

7. **Split Discogs + Beatport agrees with one:** `discogs_genres` has multiple entries, `beatport_genre` exactly equals one of them → use `beatport_genre`, confidence = **medium**. However: if a non-matching Discogs genre comes from a different family AND audio flags support that family (e.g. Discogs includes IDM and audio is `irregular` or `low_energy`), downgrade to **low** and present the alternative.

8. **Split Discogs, no Beatport:** `discogs_genres` has multiple entries, `beatport_genre` is null → use audio profile to pick:
   - If `low_energy` or `atmospheric` → prefer Downtempo-family genres if present.
   - If `dancefloor` or `high_energy` → prefer House/Techno-family genres.
   - Otherwise pick highest `style_count`. Confidence = **low**.

9. **Discogs and Beatport disagree:** `beatport_genre` NOT in `discogs_genres`:

   9a. **Same-family, different specificity** (e.g. Deep Techno vs Techno): Do NOT auto-prefer the "more specific" genre. Use audio profile to validate:
   - `atmospheric` or `low_energy` → prefer the broader/deeper variant.
   - `high_energy` → prefer the more driving/specific variant.
   - `dancefloor` with no strong audio signal → confidence = **low**, present both.

   9b. **Same-family, same depth:** Both are leaf genres → confidence = **medium**, present both.

   9c. **Cross-family + audio:** Compare candidates against audio profile. If flags clearly favor one family → use that candidate, confidence = **low**, note "audio-assisted tie-break."

   9d. **Cross-family, no audio signal:** → confidence = **insufficient**, present both.

10. **No enrichment data:** Both empty → fall through to Step D.

### Step D: Audio-only inference

When enrichment is absent. Uses audio features in order of discriminating power.

#### D.1 Broad bucket — danceability × dynamic complexity

| Danceability | Dynamic Complexity | Bucket |
|-------------|-------------------|--------|
| < 1.0 | > 10 | Ambient |
| < 1.0 | 5–10 | Experimental or Ambient |
| < 1.0 | < 5 | Downtempo or Experimental |
| 1.0–1.5 | > 5 | Downtempo or Atmospheric Techno |
| 1.0–1.5 | < 5 | Electro, IDM, or slow dance music |
| 1.5–2.5 | any | Dance music → D.2 |
| > 2.5 | any | High-energy dance → D.2 |

#### D.2 Subgenre — BPM × rhythm regularity (dance music only)

| BPM | Rhythm Regularity | Likely genres |
|-----|-------------------|---------------|
| > 155 | any | Drum & Bass, Jungle |
| 135–155 | > 0.9 | Trance, Hard Techno |
| 128–145 | > 0.9 | Techno |
| 120–135 | > 0.9 | Techno, Tech House, House |
| 118–130 | 0.8–1.0 | House, Deep House |
| 120–140 | < 0.8 | Breakbeat, Garage |
| < 120 | any | Deep House, Downtempo |

#### D.3 Spectral centroid (optional refinement)

When `spectral_centroid_mean` is available, use to distinguish within tempo-overlapping genres:

| Centroid | Tendency |
|----------|----------|
| < 600 Hz (Very Low) | Deep Techno, Dub Techno, Ambient Techno |
| 600–1200 Hz (Low) | Deep House, Minimal |
| 1200–2500 Hz (Mid) | House, Techno |
| 2500–4000 Hz (Mid-High) | Tech House, Trance |
| > 4000 Hz (High) | Hard Techno |

When null, skip — do not penalize confidence for missing centroid.

#### D.4 Confidence

- Single plausible genre → **low**, note "audio-only inference".
- Multiple plausible genres → **insufficient**, list candidates.
- Artist or label context can break ties (note "artist context") but does not raise confidence above **low**.

### Step E: Compare to current genre

11. `current` matches `suggested_genre` → **confirm** (no change).
12. `current` differs from `suggested_genre` → **conflict** (present both).
13. `current` is empty → **suggest** (propose `suggested_genre`).
14. `suggested_genre` is null → **manual** (flag for user).

### Step F: Apply session overrides

Check if user overrode any alias mapping in Step 1 that affects this track. If so, use user's preference.

## Genre Families

Used by the decision tree (Steps B.3, C.7–9) to determine same-family vs cross-family disagreements and audio veto constraints.

| Family | Genres |
|--------|--------|
| House | House, Deep House, Tech House, Afro House, Gospel House, Progressive House, Garage, Speed Garage, Disco |
| Techno | Techno, Deep Techno, Minimal, Dub Techno, Ambient Techno, Hard Techno, Drone Techno, Acid, Electro |
| Bass | Drum & Bass, Jungle, Dubstep, Breakbeat, UK Bass, Grime, Bassline, Broken Beat |
| Downtempo | Ambient, Downtempo, Dub, Dub Reggae, IDM, Experimental |
| Other | Everything not listed above |

"Same family" = both candidates in the same row. "Cross-family" = different rows or either is in Other.

## Key Mismatch Detection

Side-check during Steps 2–4. If `key_agreement` is false, surface it alongside genre results in a separate section. Not a genre decision — just useful data for the user.

## BPM Mismatch Detection

Side-check during Steps 2–4. When `bpm_agreement` is false:

1. Compute `ratio = stratum_bpm / rekordbox_bpm`.
2. If ratio is near 2.0 or 0.5 (within ±15%), it is a half-time/double-time disagreement. Use `rekordbox_bpm` for all BPM thresholds in Steps B–D — Rekordbox's analyzer has been validated by the user via beatgridding.
3. If ratio is not near a 2:1 relationship, flag as a genuine BPM conflict. Present both values to the user alongside genre results.

BPM mismatches affect audio vetoes (B.3 `slow`/`fast` flags) and subgenre inference (D.2 BPM table). An incorrect BPM can shift a track between Downtempo and Techno ranges.
